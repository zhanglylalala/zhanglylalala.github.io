15. Alert, Notification, Application Lifecycle

Posted on 2019-08-25 Edited on 2020-04-04 In Computer Science , Developing iOS App with Swift (Stanford CS193p)

Alerts and Action Sheets

Difference

The two mechanisms are all used to pop up and ask the user something.

Alerts

  1. It pops up in the middle of the screen.
  2. Usually ask questions with only two answers (e.g. OK/Cancel, Yes/No, etc.).
  3. Can be disruptive to your user-interface, so use carefully.
  4. Often used for “asynchronous” problems (“connection reset” or “network fetch failed”).
  5. Can have a text field to get a quick answer (e.g. password)
  6. Example, sometimes when Settings needs to reconfirm your Apple ID.

Action Sheets

  1. Usually slides in from the bottom of the screen on iPhone/iPod Touch, and in a popover on iPad.
  2. Can be displayed from bar button item or from any rectangular area in a view.
  3. Generally asks questions that have more than two answers.
  4. Think of action sheets as presenting “branching decisions” to the user (i.e. what next?).
  5. Example, in Safari, long press a website or use the 3D-Touch and slide up.

Create One

  1. First, declare an alert type of UIAlertController. Its initializer has three parameters.
    title is the text at the top which describes the usage of the alert.
    message is a smaller-font text beneath the title to explain the title explicitly.
    preferredStyle is which kind of alert you want. It can be .actionSheet or .alert.
  2. Then we can add actions to that alert with instance method addAction. It only takes a UIAlertAction as a parameter.
  3. To initialize a UIAlertAction, we need to pass three parameters.
    title is the name of the action showed on screen.
    style is The way we show that label of the action. It can be .cancel, .default and .destructive. The .cancel style is a bold blue one at the bottom of Action Sheets and left of Alert. .default is normal blue one arranged according to the order in which they were added. .destructive is a normal red one usually to close or delete something.
    handler is what will happen when user hit this button. This closure takes one UIAlertAction as its parameter and returns nothing.
  4. We can also add a text field with another instance method addTextField which takes one parameter called configurationHandler. It’s a closure takes a UITextField as parameter and returns nothing. In that closure, we can configure the added text field. Such as add a placeholder which we represented grayish and disappear after user typed something in. Also we can set if it’s a secure text (password or something), then it will appear each character as a dot.
  5. We can present the alert with the present method. In Action sheet of iPad, we can configure it to make it show close to the button instead of in the center. We don’t need to do these things only when the device is iPad, iPhone will automatically ignore it. And on iPad, there will be no more cancel button.

Notification

  1. Model can use Notification to communicate with Controller.
  2. Using Notification, we can find out what’s going on.
  3. For example, it can detect keyboard appeared or Document State changed or User Font Size changed.

Listen to One of Radio Station

  1. The type of observer is NSObjectProtocol.
  2. We can create an observer with NotificationCenter.default.addObserver. It has four parameters forName, object, queue and a closure.
  3. forName is the name of the radio station, it can specify the radio station we want to hear. Its type is Notification.Name. All the radio station is a static variable of Notification.Name.
  4. object is the broadcaster we want to hear. Even if something else broadcast on the same radio station, we won’t hear it.
  5. queue is OperationQueue. We can use it just like DispatchQueue, OperationQueue.main and OperationQueue.globel. It can be nil, that means that the closure will be run at the same queue as the broadcaster. We hardly assign it nil.
  6. The closure takes one parameter type of Notificatioin, it contains everything it’s broadcasting. It’s whole information is contained at the userInfo property. This closure will be push into the proper queue when broadcasts occurs.
  7. When we’re done listening, we can remove observer to stop listening.

Post A Radio Station

  1. Besides the predefined radio station, we can post our own radio station.
  2. First, we need to extense a static variable at Notification.Name. We can use its initializer Notification.Name(String) to create one.
  3. Then we need to post the radio when we want to whith NotificationCenter.default.post. It has three parameters, name, just like the forName aforementioned, is the radio station we want to post on. object is the broadcaster, usually self. userInfo is [AnyHashable : Any]? where contains any information want to pass to listeners.

KVO (Key Value Observer)

  1. KVO is also one kind of Notification. However the Notification we introduced above has a limitation. When we want to observe our own properties, sometimes we have to write our posting code several times. That’s kind of annoying. KVO is here to fix that.
  2. It’s usually used by a Controller to observe either its Model or its View.
  3. Not every property works with KVO. Only objects that inherit from NSObject can participate.
  4. To use KVO, we just need to use our instance method observe, i.e. instance.observe. It has two parameters. The first one is the property we want to observe, type of KeyPath. Its form is \Type.property or even \Type.property1.property2…. The second one is a closure, which will be excuted when that observed property is changed. It takes two parameters, the observed instance and an NSKeyValueObservedChange<Value> which contains the old value and the new value. This method returns an NSKeyValueObservation as a cookie.
  5. In KVO, we don’t removeObserver like in Notification, as long as the observation remains in the heap, the closure will stay active.
  6. Some common used KVO: UIView’s frame and center, CALayer underneath UIView, CoreData.

Application Lifecycle

Some Status

  1. Foreground Inactive: Run code without UI events. Only do something preparation.
  2. Foreground Active: Run code with receiving and processing UI events.
  3. Background: Run code without UI events for a limited time.
  4. Suspended: No code is running. Could be killed.

Transition

  1. Launch: Not running -> Foreground Inactive(for a short time) -> Foreground Active.
  2. Switch to another application: Foreground Active -> Foreground Inactive -> Background -> Suspended.
  3. Killed: Suspended -> Not Running, no code runs between suspended and killed.

Get Notified

  1. All the method mentioned is at AppDelegate.
  2. Not Running -> Foreground Inactive, application-will/didFinishLaunching will get called. And we can observe with UIApplicationDidFinishLaunching.
  3. The passed dictionary (also in notification.userInfo) tells you why your application was launched. For example, someone wants you to open a URL, you entered a certain place in the world, you are continuing an activity started on another device, a notification arrived for you (push or local), Bluetooth attached device wants to interact with you.
  4. It used to be that you would build your UI here. But nowadays we use storyboards for all that. So often you do not implement this method at all.
  5. Foreground Active -> Foreground Inactive, applicationWillResignActive will get called. And we can observe with UIApplicationWillResignActive. We can pause the UI here.
  6. Foreground Inactive -> Foreground Active, applicationDidBecomeActive will get called. And we can observe with UIApplicationDidBecomeActive. If you have paused your UI previously, here’s where you would reactivate things.
  7. Foreground Inactive -> Background, applicationDidEnterBackground will get called. And we can observe with UIApplicationDidEnterBackground.
  8. Here you want to (quickly) batten down the hatches. You only get to run for 30s or so. You can request more time, but don’t abuse this (or the system will start killing you instead). Prepare yourself to be eventually killed her (probably won’t happen, but be ready anyway).
  9. Background -> Foreground Inactive, applicationWillEnterForeground will get called. And we can observe with UIApplicationWillEnterForeground. Time to un-batten the hatches. Maybe undo what you did in DidEnterBackground.

Something Else

Other AppDelegate Items of Interest

  1. State Restoration (saving the state of your UI so that you can restore it even if you are killed).
  2. Data Protection (files can be set to be protected when a user’s device’s screen is locked).
  3. Open URL (in Xcode’s Info tab of Project Settings, you can register for certain URLs).
  4. Background Fetching (you can fetch and receive results while in the background).

UIApplication

  1. Open a URL in another application.
  2. Register to receive Push Notifications. Notifications, both local and push, are handled by the UNNotification framework.
  3. Set the fetch interval for background fetching. Usually you will set this to UIApplicationBackgroundFetchIntervalMinimum
  4. Asking for more time when backgrounded. Do NOT forget to call endBackgroundTask when you’re done!
  5. Turning on the “network in use” spinner (status bar upper left).
  6. Finding out about things.

Info.plist

  1. You can edit this file (in Xcode’s property list editor) by clicking on Info.plist
  2. Or you can even edit it as raw XML!
  3. But usually you edit Info.plist settings by clicking on your project in the Navigator.
