11&12. Drag and Drop, Table View, and Collection View, and Text Field

Posted on 2019-08-17 Edited on 2020-08-07 In 计算机科学与技术 , Swift开发 (Stanford CS193p)

Drag and Drop

  1. Drag and drop can transit data within an app in both iPhone and iPad, and can even transit from one app to the other when iPad is in multitask mode.
  2. Your app continues to work normally while drag and drop is going on.
  3. Multitouch allows some fingers to be doing drag and drop and other fingers working your app.

Create Drag

  1. It’s kind of like a “gesture recognizer” for drag and drop.
  2. We can make a view support drag and drop interaction by add an Interaction type of DragInteraction, the delegate is a subclass of UIDragInteractionDelegate that will handle the interaction, normally speaking it’s self.
  3. When the user makes a dragging gesture, the delegate gets this method. And can return the items it is willing to have dragged from the view.
  4. It returns an array because with multitouch, it is possible that user drag mutiple items at a time. Returning an empty array means “don’t drag anything after all.”
  5. As for UIDragItem, they are things like: UIDragItem(itemProvider: NSItemProvider(object: provider)). Because the data can be very expensive, we need a provider to keep UI from being blocked. So actually the data is still where it was until we perform the drop.
  6. The provider can be NSAttributedString, NSString, UIImage, NSURL, UIColor, MKMapItem, CNContact. Mark that when using the NS… types, you may want to cast them into some swift type.
  7. We can add items while dragging. If so, then the delegation gets this other function.

Create Drop

  1. Just like drag, we also need to add an Interaction to the delegate. And that delegate is a subclass of UIDropInteractionDelegate. However, compare to drag operation, the delegate of drop action needs to implement some other methods.
  2. First, we need to check if we can handle the drag item. we need to write a function to verify it. Inside the function, we can judge it using session.canLoadObjects(ofClass: NSItemProviderReading.type). We just need to put a type and .self in it.
  3. If we can handle the drag item, we can choose to the way we want with the drag items. It can be copy, move and cancel. The difference between copy and move is whether will keep the origin item. Action move could only be used in the within app drag and drop.
    We can decide this by write the dropInteraction and return the way we want.
  4. If that all goes well, then we can perform the drop. Inside that function, we will use session.loadObjects(ofClass: , completion: ). The first one just like aforementioned, pass a type.self. And the second one is a closure which requires take an array of objects as parameter and returns nothing. The passed closure will be executed some time later on the main thread. You can call multiple loadObjects(ofClass:) for different classes.

Something Else

  1. We can set session.localContext = collectionView at drag method itemsForBeginning to mark the drag item’s context.
  2. And in the drop method we can use session.localDragSession?.localContext to get the context of the drag item from. Remember that is type of Any, cast it to the type of View we want.

UITableView and UICollectionView

Basic

  1. UITableView(TV) and UICollectionView(CV) are UIScrollView subclasses used to display unbounded amounts of information.
  2. TV presents the information in a long (possibly sectioned) list. Each item will take a whole row in it.
  3. As for UICollectionView, each item only takes a grid and multiple grid formed a row.
  4. Given that the number of informations in each TV and CV is not limited, there might have millions of informations. It’s unnecessible and inefficient to create a cell for each view. In iOS, it will reuse the cells. It only create the cells that is represented on screen and one cells out, the information in that cell is replaced by the new information.

UITableView

  1. The list can be simple(first) and can be devided into sections(second).
  2. It can show simple ancillary information in Subtitle style(first), Left Detail style(second), Right Detail style(third), Basic style(above first) and arbitrarily complex information in Custom style(forth)
  3. The rows can be Grouped, but usually only when the information in the table is fixed.

UICollectionView

  1. By default, it shows the information in grids and arranges in each lines. When one line is fulled, the next grid begi`ns from next line. This way is called flow, just like text flow.
  2. Just as TV, CV can also divide into sections.
  3. Collection Views don’t have rows. They are just happen to be at the same line.
  4. We can set the scroll direction of collection view at Attributes Inspector.
  5. It’s recommanded to make the View Controller conform UICollectionViewDelegateFlowLayout.

Create UITableView and UICollectionView

Create

  1. We can drag from library and set the associated class in Identifier Inspector.
  2. There are two kinds of objects for us to use for each type, View and View Controller.
  3. If choose the View, they are merely a subview of another View.
  4. If choose the View Controller, they are the entire View and a whole new MVC.

Data

Get Data

  1. In MVC, Views cannot store datas, in the TV/CV, we set variables called dataSource and delegate, type of UITable/CollectionViewDataSource and UITable/CollectionViewDelegate in the View.
  2. The type of the dataSource var is a protocol with methods that supply the data. It knows how to ask datas from View Controller.
  3. These are automatically set for you if you use the prepackaged MVCs. However, if you drag out a UITableView or UICollectionView, you must set these vars yourself. To set it, we just need to create an outlet of view and in the didset property observer set them to self.

  4. In the View Controller, we need to override three methods to perform data retrieve.
    The first one is numberOfSections, it returns the number of sections.
    The second one is a tableView/collectionView, it returns the number of rows in a peticular section.
    The third one is another tableView/collectionView, it returns a cell to present.

Details

IndexPath
  1. IndexPath specifies which row (in TV) or item (in CV) we’re talking about.
  2. In both, you get the section the row or item is in from indexPath.section.
  3. In TV, you get which row from indexPath.row; in CV you get which item from indexPath.item.
Reuse cells
  1. In order to reuse cells, in the third method we need to do some special operation.
  2. Instead of create one cell as what we do to UILabel, UIView, we first try to take one out of the queue of presenting items.
  3. Given that we can custom the cells, we want to get the proper cells to contain the new information so we need to provide the identifier as in the segue.
  4. We also need to provide which cell we want to place the information.
  5. And if there is no spare cells for us to use(i.e. no information is getting out of the screen), it will copy a cell from a prototype with the same style as the identifier.
  6. The fact that cells are reused has serious implications for multithreading! By the time something returns from another thread, a cell might have been reused.

Styles

  1. When using the prepackaged style of UITableView, we can load up cells with things like textLabel, detailTextLabel and imageView.
  2. However, when we use the custom style in UITableView or use the UICollectionView, we have to provide the API.
  3. If we are using custom styles, we cannot use the cell we get above directly, we need to cast it into our own style class.

Custom Styles

  1. We need to drag a cell out of library put it in our View -> Prototype Cells.
  2. We will create a class which is the subclass of UITableViewCell or UICollectionViewCell
  3. We have to change its style to custom, its identity class to our own class, and modify its identifier in Attributed Inspector.
  4. We must drag some UILabels or whatever into that cell to create our own style and create outlets in our own class.
  5. As dequeueReusableCell returns a regular UITable/CollectionViewCell, if we want to use our own custom style, we have to cast the cell we get from dequeueReusableCell to our own class.

Static Table View

  1. Sometimes we just use a table view to lay out UI elements.
  2. In this case, you do not need to do any of the UITableViewDataSource stuff.
  3. You can connect outlets directly to your Controller (because there’s only one of each cell).
  4. To do this, just set your select the Table View in Document Outline and change UITableView to have Static Cells instead of Dynamic Prototypes in Attributes Inspector -> Table View -> Content.
  5. Usually static table views are Style Grouped.
  6. You can also add sections by selecting the View and modifying it at Attributed Inspector -> Table View -> Sections
  7. Add row by seleting the View Section and changing it in Attributes Inspector. You can also modify header and footer there, by the way.

Set Table View Deletable

  1. There is a predefined method called tableView(canEditRowAt indexPath). It’s down at the initial file commented. If we don’t want the view deletable, we should implement it and make it returns false.
  2. There is another method also commented call tableView(commit editingStyle) where we implement what will happen when user deletes a row.
  3. The second method can also handle the insert situation.

Segue

Extra Buttons in Table View Cell

  1. At the right end of a Table View Cell, we can add some extra buttons or symbols.
  2. We select the View at Document Outline and choose the buttons or symbols we need at Attributes Inspector -> Table View -> Accessory.

Set Segue

  1. For Table View, we can segue the view and the extra buttons separately or together.
  2. For Collection View, probably best done from this UICollectionViewDelegate method. Use performSegue(withIdentifier:) from there.
  3. Sometimes, we may need to know the section and row this cell in. So we can get the indexPath with a Table View or Collection View instance function indexPath

Update View

  1. When Model changes, we need to update View from Model. To do so, we will call reloadData
  2. This methods will causes it to call numberOfSectionsInTableView and numberOfRows/ItemsInSection all over again and then cellForRow/ItemAt on each visible row or item.
  3. That is a heavyweight method. However, if only part of Model changes, there are lighter-weight reloaders, for example

Control Some Properties

Height of Rows in a Table View Cell

  1. Row height can be fixed (UITableView’s var rowHeight: CGFloat)
  2. It can be determined using autolayout (rowHeight = UITableViewAutomaticDimension)
  3. If you do automatic, help the table view out by setting estimatedRowHeight to something
  4. The UITableView’s delegate can also control row heights.
  5. The non-estimated version of auto layout could take a lot of time.

    Size of cells in a Collection View

  6. Cell size can be fixed in the storyboard.
  7. You can also drive it from autolayout similar to table view.

  8. Or you can return the size from this delegate method

    Header and Footer for Each Section

  9. For Table View, we just supply a String for the header using this method.

  10. And for Collection View, You can’t just specify them as Strings.
  11. First you have to “turn them on” in the storyboard.
  12. They are reusable (like cells are), so you have to make a UICollectionReusableView subclass.
  13. You put your UILabel or whatever for your header, then hook up an outlet.
  14. Then you implement this dataSource method to dequeue and provide a header.
  15. Use dequeueReusableSupplementaryView(ofKind:withReuseIdentifier:for:) in there. Kind will be UICollectionElementKindSectionHeader or Footer.

Special Drag and Drop

Enable Drag

  1. The drag interaction of Collection View is unavailable in iPhone by default.
  2. We can turn it on by just set collectionView.dragInteractionEnable = true at the collectionView didSet.
  3. We can detech if a device is an iPhone or an iPad with UIDevice.current.userInterfaceIdiom. .pad means iPad.

Drop Proposal Intent

  1. In the drop method dropSessionDidUpdate of Collection View or Table View, we can not only return UICollection/TableViewDropProposal(operation: ), but also UICollection/TableViewDropProposal(operation: , intent: ). That intent parameter can receive three kinds of values .insertAtDestinationIndexPath, .insertIntoDestinationIndexPath and .unspecified.
  2. insertAtDestinationIndexPath: Insert the dropped items at the specified index path. If use this propalsal, when moving the dragged item, other items will move space for it like selection sort.
  3. insertIntoDestinationIndexPath: Dropping items is added to the drop target’s children.

Perform Drop

Coordinator

  1. Here don’t exist session. This method has a new parameter called coordinator.
  2. It has a property called destinationIndexPath. This is the drop area. It could be nil
  3. It also has another property called items, type of [UICollection/TableViewDropItem].
  4. If an item from the items is come from self, it’s property sourceIndexPath is not nil.

PerformBatchUpdate

  1. While perform Drop, maybe we need to modify the view more than once. This might crash our View. But we can use a method of that view called performBatchUpdate.
  2. It takes a closure as its first parameter. It will make the whole closure as one steps to prevent View from crashing. It can have another closure complete as animators.
  3. At the end, we have to ask coordinator drop the item to animate the drop process by calling coordinator.drop(…).
  4. We can also create a placeholder to take a place while waiting fetching the content.

Placeholder

  1. We can get a placeholder use another drop mehtod of coordinator.
  2. Later, we can load object when we fetched the item we drop using item.dragItem.itemProvider.loadObject(…). It takes an ofClass parameter like aforementioned, and a closure takes a provider and an error as parameters to load objects.
  3. When object comes, we don’t need placeholder any more. If that object fits the cell, we can call placeholder.commitInsertion(dataSourceUpdates: (IndexPath) -> Void) to replace the placeholder. In that closure, we change our model instead of View. If don’t fit, we can just use placeholderdeletePlaceholder()

UITextField

  1. Keyboard appears because UITextField becomes “first responder”. It will do this automatically when the user taps on it. Or you can make it the first responder by sending it the becomeFirstResponder message.
  2. To make the keyboard go away, send resignFirstResponder to the UITextField.
  3. UITextField has an important aspect different with UILabel. UITextField is delegate. We prefer to react to some special key with delegate. For example, when “return” key is pressed, func textFieldShouldReturn(sender: UITextField)->Bool will get called. Here is where we can react to return key. Normally return true after actions done.
  4. When the text field resigns being first responder, func textFieldDidEndEditing(sender: UITextField) get called.
  5. UITextField is a UIControl. So you can also set up target/action to notify you when things change. Just like with a button, there are different UIControlEvents which can kick off an action.
  6. Keyboards can be configured after that UITextField conformed UITextInputTraits.
  7. Keyboards can have accessory views that appear above the keyboard (custom toolbar, etc.). var inputAccessoryView:UIView
  8. The keyboard comes up over other views. So you may need to adjust your view positioning (especially to keep the text field itself visible).
  9. We can get the String user typed in the text field with textField.text
