16. More Segue

Posted on 2019-08-27 Edited on 2020-08-07 In 计算机科学与技术 , Swift开发 (Stanford CS193p)

Modal Segue

Set A Modal Segue Up

  1. Just ctrl-drag from a button to another View Controller & pick segue type “Modal”.
  2. If you need to present a Modal View Controller not from a button, use a manual segue with performSegue.
  3. If you have the View Controller itself (e.g. Alerts or from instantiateViewController), you can use the present to perform it directly.
  4. You prepare for a Modal segue just like any other segue.

Get Back from the New View Controller

  1. If there’s nothing to be said, just dismiss the segued-to MVC.
  2. To communicate results, generally you would Unwind.
  3. The presenting view controller is responsible for dismissing (not the presented). You do this by sending the presenting view controller a dismiss message, which will dismiss whatever MVC it has presented (if any).
  4. You can get at your presenting view controller with presentingViewController property.
  5. If you send this to a presented view controller, for convenience, it will forward to its presenter (unless it itself has presented an MVC, in which case it will dismiss that MVC).

Custom Presented View Controller

Appearance

  1. In horizontally regular environments, the modalPresentationStyle determines what a presented VC looks like.
  2. There are .fullScreen(default) and .overFullScreen. The first one means that the presented VC covers the presenting VC completely, while the second one means that the presented VC has a blurred background so the presenting VC is visible behind.
  3. On iPad, if full screen is “too much real estate”, there’s .formSheet and .pageSheet. These two use a smaller space and with a “grayed out” presenting view controller behind.
  4. In horizontally compact environments, these will all adapt to always be full screen.

Animation

  1. Depends on modalTransitionStyle in the view controller that is being presented.
  2. .coverVertical: slides the presented modal VC up from bottom of screen(the default)
  3. .flipHorizontal: flips the presenting view controller over to show the presented modal VC
  4. .crossDissolve: presenting VC fades out as the presented VC fades in
  5. .partialCurl: only if presenting VC is full screen and no more modal presentations coming
  6. The presentation & transition styles can be set in the storyboard by inspecting the segue.

Popover

What Does It Looks Like?

  1. Popovers pop an entire MVC over the rest of the screen.
  2. Popover’s arrow pointing to what caused it to appear.
  3. The grayed out area here is inactive. Touching in it will dismiss the popover.

Prepare for A Popover Segue

  1. All segues are managed via a UIPresentationController
  2. In a popover’s UIPopoverPresentationController, It notes what caused the popover to appear (a bar button item or just a rectangle in a view).
  3. You can also control what direction the popover’s arrow is allowed to point. Or you can control how a popover adapts to different sizes classes (e.g. iPad vs iPhone)
  4. To set those things, while preparing the segue, we need to retrieving the popoverPresentationController from the destination View Controller.
  5. We can use the popoverPresentationController we get to configure hao the popover pops up.
  6. And we can control the presentation by setting ourself (the Controller) as the delegate.

Adapt

  1. When a popover is presenting itself in a horizontally compact environment (e.g. iPhone), there might not be enough room to show a popover window comfortably, so by default it “adapts” and shows the MVC in full screen modal instead.
  2. But the popover presentation controller’s delegate can control this “adaptation” behavior.
  3. The first way is to implement method adaptivePresentationStyle. If it returns none, iOS will not adapt it, i.e. still keep popover even in iPhone.
  4. The second way is to give another MVC to represent. In this way, we usually only wrap the original MVC we want to present into a Navigation Controller or so. To do this, we need to implement the presentationController method.

Size

  1. A popover will be made pretty large unless someone tells it otherwise.
  2. The MVC being presented knows best what it’s “preferred” size inside a popover would be.
  3. It expresses that via preferredContentSize in itself (i.e. in the Controller of the MVC being presented)
  4. The MVC is not guaranteed to be that size, but the system will try its best.
  5. You can set or override the var to always return an appropriate size.
  6. Sometimes we need the size to fit a certain view. We can use the syntax view?.sizeThatFits(…). If we want to fit as small as possible, then we pass UILayoutFittingCompressedSize, or we pass UILayoutFittingExpandedSize.

Unwind

  1. This is the only segue that does not create a new MVC. It can only segue to other MVCs that (directly or indirectly) presented the current MVC.
  2. To make it work, instead of ctrl-dragging to another MVC, you ctrl-drag to the “Exit” button in the same MVC.
  3. Then you can choose a special @IBAction method you’ve created in another MVC. This means “segue by exiting me and finding a presenter who implements that method”.
  4. If no presenter (directly or indirectly) implements that method, the segue will not happen.
  5. This method must be marked @IBAction, and it must have a UIStoryboardSegue as its only argument.
  6. If the @IBAction can be found, the presented MVC will get to prepare as normal.
  7. Then the special @IBAction will be called in the other MVC and that MVC will be shown on screen.
  8. The presenting MVC will be dismissed in the process and be “unpresented” and thrown away.

Embed

  1. Putting a VC’s self.view in another VC’s view hierarchy. This can be a very powerful encapsulation technique.
  2. Drag out a Container View from the object palette into the scene you want to embed it in.
  3. Automatically sets up an “Embed Segue” from container VC to the contained VC.
